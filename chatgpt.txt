## v1.0.27 — 2026-06-15

### Ghost shop bug — root cause and fix

**Root cause**
When a shop chest was destroyed by an explosion the explosion event handlers ran at default (NORMAL) priority, removing shop blocks from the blast list. However, a plugin running at a higher priority could re-add those blocks after our handler ran, allowing the chest to be destroyed anyway. More importantly, there was no fallback cleanup path: if the physical block was removed by any means other than the owner's intentional BlockBreakEvent, the in-memory ShopManager entry, the shops.yml record, and the floating hologram all persisted. Any chest block later placed at the exact same coordinates would silently inherit the dead shop's data — the "ghost shop" scenario.

**How explosions are now handled**
- EntityExplodeEvent and BlockExplodeEvent both run at EventPriority.HIGH, giving us priority over most plugins.
- Shop blocks are removed from the blast list before the explosion processes them.
- After removing them, a one-tick delayed task re-checks each protected block. If any block is no longer a chest (meaning another plugin re-added it at HIGHEST and it was destroyed), `removeShopAt` is called immediately to purge the now-stale shop record.
- BlockBurnEvent and BlockIgniteEvent remain cancelled as before.

**How invalid shop records are cleaned up**
`removeShopAt(Location, RemovalReason)` is the single entry point for full shop deletion. It:
  1. Closes any open GUI sessions for that shop (returns items to players, fires InventoryCloseEvent)
  2. Deletes the hologram armor stands
  3. Removes the shop from ShopManager (both byOwner map and chestIndex)
  4. Deletes the entry from shops.yml and saves immediately

On plugin enable and /ps reload, `validateShops()` iterates every loaded shop record and checks whether the registered chest block is still a valid Chest block. Any shop with a missing or wrong-type block is fully removed via `removeShopInternal` before holograms are even created.

At interaction time (`ShopInteractListener`), before opening any GUI, `isGhostShop()` checks that every chest location registered to the shop still holds a chest block. If any location is stale, `removeShopAt` is called and the interaction is silently aborted.

In `ShopProtectListener.onBlockPlace`, when a player places a chest at a coordinate that is still in the chestIndex (but the previous block there was not a chest), the ghost record is purged before the placement proceeds — preventing the new chest from inheriting the old shop.

**New enum**
`data/RemovalReason` — EXPLOSION, GHOST_DETECTED, STARTUP_VALIDATION, ADMIN — passed to removal methods so the server log shows why a shop was deleted.
